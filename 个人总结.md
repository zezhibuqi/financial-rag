# Financial-RAG ai辅助项目开发个人总结报告

## 一、 项目概述

本次实训我独立完成了 `financial-rag`项目，这是一个基于检索增强生成（RAG）技术的金融财报智能问答系统。项目前端采用 Next.js + Ant Design，后端基于 Flask，数据库与向量库使用 Supabase，并创新性地引入了 BGE 向量模型与 BGE-Reranker 重排模型，实现了对茅台与宁德时代财报的精准问答。在开发过程中，我经历了从环境搭建到线上部署的完整生命周期，深刻体会到了 AI 辅助编程在工程实践中的价值与挑战。

## 二、 遇到的最大困难及解决过程

本次开发对我而言不仅是代码的编写，更是对工程化落地能力的考验。我遇到了五个主要的“拦路虎”：

**1. 技术选型的纠结**

项目初期，面对众多的技术栈组合（如 LangChain 是否引入、Embedding 模型选哪个），我感到十分迷茫。过多的选项反而导致了“决策瘫痪”。最终，我通过 TRAE 分析了各方案的优劣，决定采用轻量级的 Flask 自建 RAG 管线，放弃厚重的框架，以保证核心逻辑的透明度和可控性。

**2. 环境配置的“泥潭”**

这是最耗时且令人沮丧的阶段。在安装 `pdfplumber`和 `numpy`时，由于本地 Python 3.8 环境与 NumPy 2.0+ 的兼容性问题，以及 MinGW-w64 GCC 版本过低，导致编译失败。此外，HuggingFace 模型的下载因网络连接问题屡次中断。解决过程异常繁琐：我不得不降级 NumPy 至 1.26.4，配置系统代理环境变量，并最终改为加载本地缓存模型，才绕过了这些环境壁垒。

**3. GitHub 推送的“玄学”问题**

在代码版本管理阶段，我频繁遭遇 `Connection was reset`错误，TRAE 显示提交成功但 GitHub 上却 404。这让我意识到 IDE 内置终端可能存在的环境变量隔离问题。经过排查，我改为在系统 CMD 中使用 `git push -u origin main`命令，并重新配置了 Git 凭证，才解决了代码同步的问题。

**4. RAG 召回的准确性挑战（最核心难题）**

这是算法层面的核心困难。初期测试发现，询问“2025年茅台利润”时，模型经常回复“未找到”。深入分析发现，Embedding 模型在语义映射上存在偏差，且单纯的向量检索容易遗漏关键财务数据。为了解决跨公司对比（如“宁德时代和茅台谁利润高”）的混淆问题，我实施了**混合检索策略**：在向量检索基础上，增加了 SQL 关键词兜底查询，并构建了 `FINANCE_KW_MAP`映射表。最关键的一步是引入了 **Rerank（重排）机制**，通过 SiliconFlow 的 API 对召回片段进行二次打分。为了防止 Reranker 对长表格评分偏低，我调整了策略，采用“向量 Top-4 + 关键词保底 2 条”的混合输入方案，显著提升了答案的准确性。

**5. 线上部署的认知盲区**

本地跑通不代表项目完成。在部署至 Vercel 时，我对 Serverless 环境一无所知。遇到了依赖版本冲突、`load_dotenv`路径错误以及前端环境变量缓存等问题。通过查阅 Vercel 文档并结合 AI 指导，我学会了配置 `vercel.json`，理解了生产环境与开发环境的差异，并通过设置 Fallback URL 解决了环境变量更新延迟的问题。

## 三、 对 AI 辅助编程的心得体会

通过这次实训，我彻底改变了对编程的看法。AI（TRAE）不再是简单的“代码补全工具”，而是我的**“全知全能的结对编程伙伴”**。

**1. 从“写代码”到“审代码”的思维跃迁**

在没有 AI 的时代，初学者往往 80% 的时间花在查语法、调 Bug 上。而现在，TRAE 能瞬间生成样板代码（Boilerplate）。我的角色转变为架构师和审查员。例如在编写 Rerank 逻辑时，AI 给出了多种方案，我需要理解每种方案的利弊（如方案 A 的语义丢失风险 vs 方案 B 的性能开销），并做出决策。这种“指挥 AI 干活，人类把控方向”的模式，极大地提升了开发上限。

**2. 精准的“翻译官”与“调试器”**

在环境配置和部署阶段，AI 发挥了巨大作用。面对晦涩的 GCC 编译错误或无响应的 Git 推送，AI 能迅速定位问题根源（如版本冲突、网络代理），并给出精确的修复指令。它就像一本活的《Stack Overflow》，不仅能给答案，还能结合我的具体上下文解释原因，帮助我真正理解了背后的原理，而不仅仅是复制粘贴。

**3. 突破知识边界的“导师”**

在处理 RAG 的高级优化（如 Rerank 策略调整）时，如果靠自己啃论文，可能需要一周。但 AI 能将复杂的算法逻辑转化为可执行的代码策略。它教会了我如何构建关键词映射表，如何处理长文本的截断问题。这种“做中学”的效率是传统教学模式无法比拟的。

**4. 警惕“幻觉”，保持批判性思维**

当然，AI 并非万能。在开发过程中，AI 也曾给出过错误的导入路径或过时的 API 调用方式。这让我深刻认识到，信任但验证（Trust but Verify）是使用 AI 的核心准则。特别是在金融数据这种容错率极低的场景下，必须对 AI 生成的每一段逻辑、每一个计算公式进行严格的测试（Golden Set 测试），绝不能盲目采信。

**结语**

这次项目让我明白，未来的程序员核心竞争力不在于记忆多少 API，而在于能否清晰地定义问题、准确地向 AI 下达指令（Prompt Engineering），以及对生成结果的鉴别与优化能力。AI 辅助编程不是“偷懒”，而是将人类的创造力从繁琐的语法中解放出来，去解决更有价值的逻辑难题。
