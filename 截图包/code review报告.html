1. RAG 管线提取为深度模块

Strong
Files

backend/app.py (380行) — /api/chat 路由函数包含完整 RAG 管线

Problem

当前 chat() 函数是一个浅模块：接口（HTTP 路由）和实现（RAG 管线逻辑）紧耦合。约 180 行的 RAG 逻辑（Embedding、向量检索、关键词补充、Rerank、Prompt 组装、LLM 生成）直接内联在路由函数中。这导致：

  • 可测试性差：无法独立测试 RAG 管线，必须启动 Flask 服务器
  • 复用性差：如果未来需要 CLI 工具或批量处理，RAG 逻辑无法复用
  • 认知负荷高：理解 /api/chat 需要同时理解 HTTP 层和 RAG 领域
Solution

提取 RAGPipeline 类或 rag.py 模块，封装完整的检索-增强-生成流程。接口简化为 answer(question, company?, year?) -> {answer, sources}。HTTP 路由只负责参数解析和响应序列化。

    A -->|包含| C[向量检索]
    A -->|包含| D[关键词补充]
    A -->|包含| E[Rerank]

    B -->|封装| E[Rerank]
    B -->|封装| F[生成]
    G[CLI / 测试] -->|调用| B
Benefits
  • Locality：RAG 逻辑变更集中在 rag.py，HTTP 层稳定
  • Leverage：测试可直接调用 RAGPipeline.answer()，无需 HTTP 开销
  • Deletion test：删除 rag.py 后，HTTP 路由变成空壳，证明它是复杂性的集中点

2. 关键词补充逻辑提取为领域模块

Strong
Files

backend/app.py 第 153-219 行 — FINANCE_KW_MAP 和关键词检索逻辑

Problem

关键词补充是财报领域的核心知识（9 类财务指标映射），但直接硬编码在路由函数中。这导致：

  • 领域知识散落：财务指标映射是领域概念，不应与 HTTP 逻辑混在一起
  • 难以演进：新增指标类型需要修改路由函数，而非独立的领域模块
  • 无法独立测试：关键词触发逻辑、公司提取逻辑无法单独验证
Solution

提取 keyword_supplement.py 模块，接口为 supplement(question, vector_results) -> list[Chunk]。内部封装 FINANCE_KW_MAP、公司/年份提取、Supabase LIKE 查询。

3. 前端 API Client 统一

Worth exploring
Files

frontend/pages/index.tsx, frontend/pages/manage.tsx — 分散的 fetch 调用

Problem

两个页面各自硬编码 API_BASE 和 fetch 逻辑。如果后端地址变更，需要修改多处。错误处理（message.error）也重复。

4. ingest.py 的 process_pdf 返回值重构

Speculative
Files

backend/ingest.py 第 229-281 行 — process_pdf 函数

Problem

process_pdf 返回 (all_chunks, company, year, filename) 四元组。调用者（main 函数第 312-313 行）需要记住位置：all_chunks, company, year, filename = result。这是浅接口。

Solution

返回 IngestResult dataclass 或 dict：{chunks, company, year, doc_name}。调用者通过属性名访问，而非位置。

Benefits
  • 可读性：result.chunks 比 result[0] 更清晰
  • 扩展性：新增字段（如 warnings）不破坏调用者

5. 测试基础设施统一

Speculative
Files

backend/test/*.py — 9 个测试脚本

Problem

每个测试脚本都独立构造 HTTP 请求、解析响应、打印结果。重复代码多，且无法批量运行或生成报告。

Solution

提取 test_base.py，提供 run_chat_test(name, body, expect_fn) 辅助函数。golden_set.py 已经是这个方向，可以扩展为通用框架。

Benefits
  • Locality：测试逻辑变更集中在 test_base.py
  • 批量运行：可一键运行所有测试

Top Recommendation

候选 1: RAG 管线提取为深度模块

这是最值得优先探索的改进。app.py 的 /api/chat 路由函数承载了过多职责，是典型的"浅模块"——接口（HTTP 路由）和实现（RAG 管线）几乎一样复杂。提取 RAGPipeline 后：

  • HTTP 层变成薄适配层，职责清晰
  • RAG 逻辑可独立测试，无需启动 Flask
  • 未来扩展（如流式响应、批量处理）有清晰的扩展点

这个改进符合"深度模块"原则：一个小的接口（answer()）背后是大量复杂行为。
